### **项目简介：SuiStride 是什么？ (Slide 3: 产品截图 - 跑步界面/奖金池列表)**

SuiStride 是一款构建在 Sui 网络上的 Run-to-Earn dApp。用户可以通过质押（Stake）SUI 或我们的原生Token，加入到一个有时限的跑步奖金池中。在活动期间，通过我们的移动 App 记录真实有效的跑步数据，最终与奖金池中的其他参与者一较高下，根据排名分享丰厚的奖金。

它不仅仅是一个跑步记录工具，更是一个结合了 GameFi 和 SocialFi 的 Web3 健身平台。

### **我们如何脱颖而出？(Slide 4: 三大核心优势图标)**

我们解决了 Move-to-Earn 赛道的两大核心痛点：**用户体验** 和 **数据可信度**。

1.  **无感知的 Web3 体验 - ZKLogin 的力量**:
    *   **(用户痛点)**: “我奶奶也想跑步赚钱，但她搞不定钱包和助记词。”
    *   **(我们的解决方案)**: 我们深度集成了 Sui 的原生 ZKLogin 功能。新用户无需下载钱包、无需抄写助记词，直接使用他们已有的 Google 账户即可一键登录。在后端，Enoki 服务会为他们生成一个与该账户关联的链上地址和密钥对。**这意味着，对于用户来说，SuiStride 的注册登录流程和任何一款 Web2 App 毫无区别。** 这为我们开启了触达海量圈外用户的大门。

2.  **高效、低成本的防作弊机制**:
    *   **(用户痛点)**: “我每天辛苦跑10公里，却比不过在家里晃动手机的作弊者。”
    *   **(我们的解决方案)**: 我们设计了一套精密的多层数据验证系统，确保“所跑即所得”。这套系统的核心思想是：**链下精细处理，链上高效承诺**。我会在后面的技术部分详细拆解。

3.  **快速迭代的跨平台架构**:
    *   **(行业痛点)**: 很多项目要么只有网页版，要么移动端功能落后。
    *   **(我们的解决方案)**: 我们采用 React Native + WebView 的混合开发模式，Web 与 Mobile 共享超过95%的代码。这让我们能以极高的效率同步迭代功能，确保无论是网页用户还是手机用户，都能在第一时间体验到最新、最完整的功能。

### **技术深度剖析：我们如何构筑信任？ (Slide 5: 防作弊流程图)**

现在，让我们深入探讨 SuiStride 最硬核的部分：如何确保每一步都真实可信。

**第一层：客户端数据清洗与预处理**

我们从不信任原始的 GPS 数据。

-   **(平滑降噪)**: 首先，我们通过一个**卡尔曼滤波器(Kalman Filter)**对设备采集到的经纬度坐标进行实时处理。这个滤波器能有效剔除因信号遮挡或设备问题产生的异常“漂移点”，输出一条平滑且更接近真实物理世界的运动轨迹。
-   **(隐私保护与分段)**: 接着，我们将绝对坐标转换为相对坐标，并按1分钟为单位，将轨迹切割为多个 `RunSegment`（运动分片）。每个分片都是对这一分钟运动的快照，包含了平均/最大速度、步数、加速度方差等关键指标。这不仅保护了用户隐私，也为后续的验证提供了标准化数据单元。

**第二层：用密码学“指纹”锁定真相 - 默克尔树**

将所有分片数据直接上传到区块链，Gas 费将是天文数字。我们的解决方案是默克尔树。

-   **(序列化与哈希)**: 每个 `RunSegment` 被确定性地序列化成一段二进制数据。然后，我们对每个分片的二进制数据进行 SHA-256 哈希，得到叶子节点。
-   **(构建承诺)**: 我们将这些哈希两两组合，再次哈希，层层向上，最终构建出一棵完整的默克尔树，并得到唯一的**默克尔根 (Merkle Root)**。这个 32 字节的根，就是对用户本次跑步所有详细数据的、不可篡改的密码学承诺。

**第三层：轻量级的链上最终验证**

-   **(高效上链)**: 跑步结束时，用户提交到 Sui 网络的，**不是庞大的轨迹数据，而仅仅是这个默克尔根**。
-   **(智能合约)**: 我们的 `core.move` 合约中的 `submit_run_with_proof` 函数接收这个根，并将其与用户的步数一同记录在 `StakingPool` 的参与者信息中。同时，为了防止“重放攻击”（即重复提交同一次跑步数据），`start_run` 时创建的 `Session` 对象会在 `submit` 时被立即销毁。这个 `Session` 对象包含了一个与当前纪元（Epoch）关联的 nonce，确保了每次跑步都是一次性的。

**(小结)**: 这套机制的优越性在于：
-   **极低成本**: 将大量计算和数据存储放在链下，用户仅需为一次交易（存储一个哈希）支付 Gas。
-   **可扩展性**: 无论用户跑1公里还是42公里，上链的成本是恒定的。
-   **可验证性**: 虽然完整数据不上链，但我们保留了事后验证的能力。任何对结果的质疑，都可以通过链下的验证者程序，用默克尔证明来公开、透明地裁决。

### **架构概览 (Slide 6: 整体架构图)**

**智能合约 (Move):**

-   我们的核心是 `StakingPool` 共享对象，它像一个公开的竞技场。`UserData` 和 `Session` 对象则归属于用户，保证了用户对其个人数据和当前活动状态的所有权。这种对象模型的设计，完美体现了 Sui 的特性。

**后端 (隐式):**

-   虽然我们是 dApp，但 ZKLogin 依赖 Enoki 服务。同时，未来我们也会构建自己的链下验证和数据分析服务。

**前端 (TypeScript/React):**

-   **Web/WebView (Next.js)**: 作为核心 UI 和业务逻辑层，负责与 Sui 区块链交互（通过 `@mysten/dapp-kit` 和 `SuiGrpcClient`），执行所有防作弊计算，并通过 `postMessage` 与原生移动端外壳通信。
-   **原生移动端外壳 (React Native)**: 它的职责非常纯粹——提供 WebView 容器，并作为 Web 内容与设备硬件之间的“桥梁”。它调用系统级的 GPS 服务，并将数据高效地传递给 Web 层；同时，它也负责拉起系统浏览器以完成 ZKLogin 的 OAuth 流程。
